# Agentic Workflow

This repository is designed for issue-scoped agent work. Claude, Codex, and other coding agents can be useful here, but they must operate inside explicit guardrails because the site is backed by production Supabase data.

## Default Rule

Start from an issue. If no issue exists, the first agent task is to draft one using the structured templates, not to mutate code or data.

Agents should classify every task as one or more of:

- `code`: React, Next.js, Tailwind, shadcn/ui, loaders, validation, tests
- `content`: `pages`, `sections`, `entities`, `section_entities`, `entity_relations`
- `schema`: tables, columns, indexes, functions, triggers, RLS, storage policies
- `ops`: Vercel, Supabase Auth, SMTP, domains, environment variables
- `research`: YouTube, Instagram, history PDF, references, visual direction

## Allowed Agent Work

Agents may do these without additional approval after an issue exists:

- Inspect files and schemas.
- Propose implementation plans.
- Edit code within the issue scope.
- Add migrations for review.
- Add data migrations that upsert content by stable keys such as `slug` or `key`.
- Run read-only Supabase queries through MCP.
- Run local quality checks.
- Draft GitHub issues using `.github/ISSUE_TEMPLATE`.

## Production Write Policy

Supabase production writes require maintainer approval. Approval must be explicit in the issue, PR, or chat, and the agent must state the exact command/tool it will use.

Acceptable production writes:

- Applying a committed migration with `supabase db push`.
- Applying the SQL from a committed migration through MCP `apply_migration`.
- Running a reviewed one-off maintenance script that is already in `scripts/`.

Not acceptable:

- Applying SQL that only exists in chat.
- Applying SQL generated by an agent without a migration file.
- Fixing production data with ad-hoc dashboard or MCP writes and no record.
- Using the service role key as a shortcut for normal user flows.

## Hard Guardrails

Agents must stop and ask for maintainer approval before any command or SQL containing:

- `drop schema`
- `drop table`
- `truncate`
- `delete from` without a narrow `where`
- `update` without a narrow `where`
- `alter table ... disable row level security`
- `drop policy`
- `drop function`
- `drop trigger`
- `delete storage.objects`
- bucket deletion or bucket visibility changes
- `supabase db reset` against anything other than local development

Agents must never:

- Commit secrets.
- Put a service role key in `NEXT_PUBLIC_*`.
- Add service role usage to browser/client code.
- Disable RLS to make a query work.
- Rewrite the content graph structure without a schema issue and migration review.
- Remove fallback failures by hardcoding production content in React.
- Change Auth redirect URLs, SMTP, domains, or Vercel env without an ops issue.

## Supabase MCP Rules

MCP is read-only by default.

Safe MCP actions:

- `list_tables`
- `list_migrations`
- `execute_sql` for `select` queries
- `get_advisors`
- `generate_typescript_types`
- `search_docs`

Restricted MCP actions:

- `apply_migration`
- storage policy changes
- auth-related DDL
- trigger/function changes
- any insert/update/delete against production content

When using `apply_migration`, the agent must:

1. Point to the migration file path.
2. Confirm the SQL was read from that file.
3. Confirm no hard-guardrail statement is present.
4. Apply exactly that SQL.
5. Record the migration name in the PR or final summary.

## Issue Drafting Pattern

When asked to "make an issue" or "organize the task", agents should produce a GitHub-ready issue.

Minimum fields:

- Title
- Type
- User-facing goal
- Affected routes
- Supabase impact
- Code impact
- Acceptance checks
- Non-goals
- Risk/guardrail notes

Use `.agents/skills/create-bremen-issue/SKILL.md` when available.

## Implementation Pattern

For normal code tasks:

1. Read the issue.
2. Inspect only the relevant files, migrations, and loaders.
3. State the write scope before editing.
4. Make the smallest coherent change.
5. Run relevant checks.
6. Summarize changed behavior and remaining risk.

For content graph tasks:

1. Identify page slug and section keys.
2. Determine whether this is content data, renderer behavior, or schema.
3. Prefer upserts by stable keys (`pages.slug`, `sections.key`, `entities.slug`).
4. Link with `section_entities` or `entity_relations`.
5. Verify counts with the query in `docs/content-graph.md`.

For schema tasks:

1. Create a migration.
2. Keep DDL explicit and minimal.
3. Preserve RLS.
4. Regenerate `lib/supabase/types.ts`.
5. Run Supabase advisors after applying.

## Final Response Requirements

Agent final summaries should include:

- What changed.
- Whether Supabase changed.
- Migration file names, if any.
- Checks run.
- Checks not run and why.
- Any production actions taken.

Do not hide skipped checks. Do not imply production data was changed if only local files were edited.
